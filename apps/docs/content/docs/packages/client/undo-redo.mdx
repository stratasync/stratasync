---
title: Undo and redo
description: Client-side undo and redo with full transaction history.
---

The sync client tracks an undo/redo stack so you can reverse and replay local mutations. Every create, update, delete, archive, and unarchive operation records a history entry.

## Basic usage

Call `undo()` and `redo()` after any mutation to walk the history stack.

```ts
await client.create("Task", { title: "Task 1", status: "todo" });
await client.update("Task", "task-123", { title: "Updated" });

if (client.canUndo()) {
  await client.undo();
}

if (client.canRedo()) {
  await client.redo();
}
```

## API reference

These methods are available on the sync client instance.

### canUndo()

Returns `true` if the undo stack has entries.

```ts
const canUndo: boolean = client.canUndo();
```

### canRedo()

Returns `true` if the redo stack has entries.

```ts
const canRedo: boolean = client.canRedo();
```

### undo()

Pops the last entry from the undo stack, applies the inverse operation, and pushes the entry to the redo stack.

```ts
await client.undo();
```

If the undo operation fails (due to a conflict, for example), the client pushes the entry back onto the undo stack and throws the error.

### redo()

Pops the last entry from the redo stack, re-applies the original operation, and pushes the entry back to the undo stack.

```ts
await client.redo();
```

## How it works

Each mutation creates a history entry containing both the inverse (undo) and original (redo) operations. Calling `undo()` executes the inverse as a new transaction through the outbox, with history recording suppressed so it doesn't create its own entry. `redo()` follows the same process in reverse.

See [Transactions](/docs/packages/core/transactions) for the complete action-to-inverse mapping.

## History and the outbox

Undo/redo operations create real transactions that flow through the outbox to the server — they aren't client-only rollbacks.

- The client persists and syncs undo operations like regular mutations
- Other connected clients see the undo effect as a normal update, delete, or insert
- If the server rejects a transaction, the client removes the corresponding history entry from both stacks

## Stack behavior

The stacks follow standard undo/redo semantics.

- Making a new mutation clears the redo stack
- Calling `clearAll()` clears both stacks
- The client removes history entries if the server rejects their associated transaction

## React integration

Use the sync client hooks in `@stratasync/react` to bind undo/redo to your UI.

```tsx
import { useSyncClientInstance } from "@stratasync/react";

function UndoRedoButtons() {
  const client = useSyncClientInstance();

  return (
    <div>
      <button onClick={() => client.undo()} disabled={!client.canUndo()}>
        Undo
      </button>
      <button onClick={() => client.redo()} disabled={!client.canRedo()}>
        Redo
      </button>
    </div>
  );
}
```

## Keyboard shortcuts

Bind `Cmd/Ctrl+Z` and `Cmd/Ctrl+Shift+Z` to undo and redo.

```tsx
import { useEffect } from "react";
import { useSyncClientInstance } from "@stratasync/react";

function useUndoRedoShortcuts() {
  const client = useSyncClientInstance();

  useEffect(() => {
    const handler = (e: KeyboardEvent) => {
      if (!(e.metaKey || e.ctrlKey)) return;

      if (e.key === "z" && !e.shiftKey) {
        e.preventDefault();
        client.undo();
      } else if (e.key === "z" && e.shiftKey) {
        e.preventDefault();
        client.redo();
      }
    };

    window.addEventListener("keydown", handler);
    return () => window.removeEventListener("keydown", handler);
  }, [client]);
}
```

---
title: Configuration
description: SyncClientOptions reference for configuring the Strata Sync client.
---

The `createSyncClient` factory accepts a `SyncClientOptions` object that configures storage, transport, reactivity, and sync behavior.

## SyncClientOptions

```ts
import { createSyncClient } from "@stratasync/client";

const client = createSyncClient({
  // Required adapters
  storage: storageAdapter,
  transport: transportAdapter,
  reactivity: reactivityAdapter,

  // Optional configuration
  schema: schemaDefinition,
  modelFactory: customFactory,
  dbName: "my-app-sync",
  userId: "user-123",
  version: 1,
  userVersion: 1,
  groups: ["workspace-abc"],
  optimistic: true,
  batchMutations: true,
  batchDelay: 50,
  bootstrapMode: "auto",
  yjsTransport: yjsTransportAdapter,
  rebaseStrategy: "server-wins",
  fieldLevelConflicts: true,
});
```

## Required options

### storage

**Type:** `StorageAdapter`

The storage adapter for local persistence. It handles model rows, metadata, outbox, partial indexes, and sync actions.

```ts
import { createIndexedDbStorage } from "@stratasync/storage-idb";

const storage = createIndexedDbStorage();
```

The `StorageAdapter` interface defines methods for:

- `get`, `getAll`, `put`, `delete` -- basic CRUD for model rows
- `getByIndex` -- indexed lookups
- `getMeta`, `setMeta` -- sync metadata (last sync ID, schema hash, and more)
- `getOutbox`, `addToOutbox`, `removeFromOutbox` -- pending transaction persistence
- `hasPartialIndex`, `setPartialIndex` -- partial model index tracking
- `addSyncActions`, `getSyncActions`, `clearSyncActions` -- sync action log
- `writeBatch` -- batch write operations
- `clear`, `count` -- storage management

### transport

**Type:** `TransportAdapter`

The transport adapter for server communication. It handles bootstrap, delta subscriptions, batch mutations, and connection state.

```ts
import { createGraphQLTransport } from "@stratasync/transport-graphql";

const transport = createGraphQLTransport({
  endpoint: "https://api.example.com/graphql",
  syncEndpoint: "https://api.example.com/sync",
  wsEndpoint: "wss://api.example.com/sync",
  auth: { getAccessToken: async () => "token" },
});
```

The `TransportAdapter` interface defines:

- `bootstrap(options)` -- stream initial data as `AsyncGenerator<ModelRow>`
- `batchLoad(options)` -- lazy-load models by index
- `mutate(batch)` -- submit transaction batches
- `subscribe(options)` -- real-time delta subscription
- `fetchDeltas(after, limit?)` -- poll for missed deltas
- `getConnectionState()` -- current connection state
- `onConnectionStateChange(callback)` -- subscribe to connection changes
- `close()` -- clean up resources

### reactivity

**Type:** `ReactivityAdapter`

The reactivity adapter for observable state. It makes identity map entries reactive so UI frameworks re-render on changes.

```ts
import { createMobXReactivity } from "@stratasync/mobx";

const reactivity = createMobXReactivity();
```

## Optional options

### schema

**Type:** `SchemaDefinition | ModelRegistrySnapshot`

Explicit schema definition. If not provided, the client uses the global `ModelRegistry` populated by decorators.

```ts
const client = createSyncClient({
  schema: {
    models: {
      Task: {
        loadStrategy: "instant",
        fields: { title: {}, status: {} },
      },
    },
  },
  // ...
});
```

### modelFactory

**Type:** `ModelFactory | ModelFactoryFactory`

Custom factory for creating model instances from raw data. By default, the client uses the `ModelRegistry` to find constructors and then creates instances with `_applyUpdate`.

```ts
// Simple factory
const factory = (modelName: string, data: Record<string, unknown>) => {
  return { ...data, __model: modelName };
};

// Factory with store context
const factoryFactory = ({ store }) => {
  return (modelName: string, data: Record<string, unknown>) => {
    const instance = new (ModelRegistry.getModelConstructor(modelName))();
    instance.store = store;
    instance._applyUpdate(data);
    return instance;
  };
};
```

### dbName

**Type:** `string` -- **Default:** `"sync-db"`

Database name for storage. Each unique name creates a separate IndexedDB database.

### userId

**Type:** `string`

Logged-in user ID. The client uses this for storage isolation in multi-user scenarios.

### version / userVersion

**Type:** `number`

Client version and per-user version for storage naming. Incrementing these values triggers a fresh bootstrap.

### groups

**Type:** `string[]`

Sync groups for multi-tenancy. The client syncs only models belonging to the specified groups.

```ts
const client = createSyncClient({
  groups: ["workspace-abc", "workspace-def"],
  // ...
});
```

### optimistic

**Type:** `boolean` -- **Default:** `true`

Enable optimistic updates. When `true`, the client applies mutations to the identity map immediately before sending them to the server. If the server rejects the mutation, the client rolls back the change.

### batchMutations

**Type:** `boolean` -- **Default:** `false`

Batch multiple mutations together before sending them to the server. This is useful for reducing network requests when the user makes rapid changes.

### batchDelay

**Type:** `number` (milliseconds)

Delay before sending a mutation batch. Only applies when `batchMutations` is `true`.

### bootstrapMode

**Type:** `"auto" | "full" | "local"` -- **Default:** `"auto"`

Controls how the client initializes:

| Mode      | Behavior                                                                              |
| --------- | ------------------------------------------------------------------------------------- |
| `"auto"`  | Uses local storage if available and schema matches, otherwise bootstraps from server. |
| `"full"`  | Always bootstraps from the server, clearing local storage.                            |
| `"local"` | Only uses local storage. Never contacts the server for bootstrap.                     |

### yjsTransport

**Type:** `YjsTransport`

Optional Yjs transport for collaborative document editing. When you provide this, the client creates `YjsDocumentManager` and `YjsPresenceManager` instances accessible via `client.yjs`.

### rebaseStrategy

**Type:** `"server-wins" | "client-wins" | "merge"` -- **Default:** `"server-wins"`

Default conflict resolution strategy for transaction rebasing when the server has concurrent changes:

| Strategy        | Behavior                                                                |
| --------------- | ----------------------------------------------------------------------- |
| `"server-wins"` | Server changes take precedence. Client changes are dropped on conflict. |
| `"client-wins"` | Client changes take precedence. Server changes are overridden.          |
| `"merge"`       | Field-level merge. Non-conflicting fields from both sides are kept.     |

### fieldLevelConflicts

**Type:** `boolean` -- **Default:** `true`

Enable field-level conflict detection during rebasing. When `true`, the client detects conflicts per-field rather than per-model, allowing non-overlapping field changes to coexist.

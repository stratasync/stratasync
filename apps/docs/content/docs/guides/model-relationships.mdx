Strata Sync models support rich relationships through decorators. This guide covers every relationship type with complete examples, including lazy loading, circular references, and a full domain model.

## Relationship decorators at a glance

| Decorator              | Direction            | Cardinality  | Creates foreign key?      |
| ---------------------- | -------------------- | ------------ | ------------------------- |
| `@Reference`           | Model A -> Model B   | One-to-one   | Yes (on A)                |
| `@ManyToOne`           | Model A -> Model B   | Many-to-one  | Yes (on A)                |
| `@OneToMany`           | Model B -> Model A[] | One-to-many  | No (uses A's foreign key) |
| `@BackReference`       | Computed inverse     | Varies       | No                        |
| `@ReferenceArray`      | Model A -> Model B[] | Ordered list | Yes (array field on A)    |
| `@ReferenceCollection` | Alias for @OneToMany | One-to-many  | No                        |

## @Reference -- belongs to

`@Reference` creates a one-to-one relationship where the current model holds the foreign key. It is equivalent to a "belongs to" relationship in traditional ORMs.

```ts
import { Model, ClientModel, Property, Reference } from "@stratasync/core";

@ClientModel("Task", { loadStrategy: "instant" })
export class Task extends Model {
  @Property()
  declare id: string;

  @Property()
  declare title: string;

  // Creates an `assigneeId` foreign key property automatically
  @Reference(() => User, "assignedTasks")
  declare assignee: User | null;

  // The foreign key is available as a regular property
  @Property()
  declare assigneeId: string | null;
}
```

### How it works

The `@Reference` decorator:

1. Takes a factory function `() => ModelConstructor` as its first argument (to handle circular imports).
2. Optionally takes an inverse property name as the second argument.
3. Automatically creates a foreign key property named `${propertyName}Id` (for example, `assignee` creates `assigneeId`).
4. Registers both the reference model property and the foreign key property with the model registry.
5. Sets up a cached getter so that accessing `task.assignee` resolves the referenced model from the identity map.

### Custom foreign key

Override the default foreign key name with the `foreignKey` option:

```ts
@Reference(() => User, "createdTasks", { foreignKey: "creatorId" })
declare creator: User | null;

@Property()
declare creatorId: string | null;
```

### Options

| Option       | Type      | Default     | Description                       |
| ------------ | --------- | ----------- | --------------------------------- |
| `foreignKey` | `string`  | `${name}Id` | Name of the foreign key property  |
| `nullable`   | `boolean` | --          | Whether the reference can be null |
| `indexed`    | `boolean` | auto        | Whether to index the foreign key  |
| `lazy`       | `boolean` | --          | Whether to lazily hydrate         |

## @ManyToOne -- explicit many-to-one

`@ManyToOne` is functionally identical to `@Reference` but accepts a model name string instead of a factory function, making it more explicit about the relationship direction.

```ts
import { Model, ClientModel, Property, ManyToOne } from "@stratasync/core";

@ClientModel("Comment", { loadStrategy: "lazy" })
export class Comment extends Model {
  @Property()
  declare id: string;

  @Property()
  declare body: string;

  // String-based model name (no factory function needed)
  @ManyToOne("Task", "comments", { foreignKey: "taskId" })
  declare task: unknown;

  @Property()
  declare taskId: string;

  // Also supports factory functions
  @ManyToOne(() => User, "comments", { foreignKey: "authorId" })
  declare author: unknown;

  @Property()
  declare authorId: string;
}
```

### When to use @ManyToOne vs @Reference

- Use `@ManyToOne` when you want to be explicit about the string model name or want to avoid the factory function pattern.
- Use `@Reference` when you prefer the factory function pattern (which helps with circular imports in TypeScript).

Both decorators produce identical metadata and runtime behavior.

## @OneToMany -- inverse collection

`@OneToMany` defines the inverse side of a many-to-one relationship. The decorator creates a `LazyCollection` that resolves all models pointing to the current instance via their foreign key.

```ts
import { Model, ClientModel, Property, OneToMany } from "@stratasync/core";

@ClientModel("Project", { loadStrategy: "instant" })
export class Project extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  // All tasks where task.projectId === this.id
  @OneToMany({ foreignKey: "projectId" })
  declare tasks: unknown[];
}
```

### How LazyCollection works

The `@OneToMany` decorator creates a `LazyCollection` that:

1. On first access, attaches itself to the parent model instance.
2. Queries the identity map for all models matching the foreign key.
3. Caches the result and updates reactively when you add or remove models.

```ts
// Accessing the collection triggers resolution
const project = client.getCached("Project", projectId);
const tasks = project.tasks; // LazyCollection<Task>

// The collection is iterable
for (const task of tasks) {
  // ...
}
```

### Options

| Option       | Type      | Default | Description                        |
| ------------ | --------- | ------- | ---------------------------------- |
| `foreignKey` | `string`  | --      | Foreign key on the child model     |
| `indexed`    | `boolean` | --      | Whether to use index for lookups   |
| `nullable`   | `boolean` | --      | Whether the collection can be null |

## @BackReference -- computed inverse

`@BackReference` creates a computed inverse of a reference relationship. Unlike `@OneToMany`, the decorator doesn't create a collection -- it represents the inverse lookup for documentation and tooling purposes.

```ts
import { Model, ClientModel, Property, BackReference } from "@stratasync/core";

@ClientModel("User", { loadStrategy: "instant" })
export class User extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  // Computed inverse of Task.assignee
  @BackReference({ foreignKey: "assigneeId" })
  declare assignedTasks: unknown[];

  // Computed inverse of Comment.author
  @BackReference({ foreignKey: "authorId" })
  declare comments: unknown[];
}
```

### Options

| Option       | Type     | Default | Description                          |
| ------------ | -------- | ------- | ------------------------------------ |
| `foreignKey` | `string` | --      | Foreign key on the referencing model |

## @ReferenceArray -- ordered list of references

`@ReferenceArray` stores an ordered list of references to other models. The decorator preserves the order in the array field on the model.

```ts
import { Model, ClientModel, Property, ReferenceArray } from "@stratasync/core";

@ClientModel("Board", { loadStrategy: "instant" })
export class Board extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  // Ordered list of column IDs
  @ReferenceArray()
  declare columns: unknown[];

  @Property()
  declare columnIds: string[];
}
```

### Through model

For many-to-many relationships, use the `through` option to specify a join model:

```ts
@ReferenceArray({ through: "ProjectMember" })
declare members: unknown[];
```

### Options

| Option    | Type     | Default | Description                      |
| --------- | -------- | ------- | -------------------------------- |
| `through` | `string` | --      | Join model name for many-to-many |

## @ReferenceCollection -- unordered set

`@ReferenceCollection` is an alias for `@OneToMany`. You use it when you want to emphasize that the collection is unordered:

```ts
import {
  Model,
  ClientModel,
  Property,
  ReferenceCollection,
} from "@stratasync/core";

@ClientModel("Team", { loadStrategy: "instant" })
export class Team extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  // Unordered set of team members
  @ReferenceCollection({ foreignKey: "teamId" })
  declare members: unknown[];
}
```

## Lazy loading vs. eager loading

By default, the client resolves references eagerly from the identity map when accessed. You use the `lazy` option to defer hydration until you explicitly need the referenced model.

```ts
// Eager (default): resolved from identity map on access
@Reference(() => User, "assignedTasks")
declare assignee: User | null;

// Lazy: not hydrated until explicitly accessed
@Reference(() => User, "assignedTasks", { lazy: true })
declare assignee: User | null;
```

### When to use lazy loading

- **Large reference chains**: If model A references B, which references C, which references D, eager loading could pull in a large graph on first access. Use `lazy: true` on less-frequently-accessed references.
- **Partial load models**: If the referenced model uses `loadStrategy: "partial"` or `"explicitlyRequested"`, the reference must be lazy since the data may not be available yet.
- **Performance**: Lazy references avoid the cost of resolving references for models that are loaded but never displayed.

## Circular reference handling

The identity map and the factory function pattern handle circular references (A references B, B references A) naturally:

```ts
@ClientModel("Task", { loadStrategy: "instant" })
export class Task extends Model {
  @Property()
  declare id: string;

  @Reference(() => Task, undefined, { foreignKey: "parentId" })
  declare parent: Task | null;

  @Property()
  declare parentId: string | null;

  @OneToMany({ foreignKey: "parentId" })
  declare children: unknown[];
}
```

Key considerations:

- The factory function `() => Task` is required because `@Reference` calls `modelFactory()` eagerly at decoration time to resolve the model name. The factory pattern works because TypeScript hoists class declarations, so even though the factory is invoked immediately, the class is already available. Without the factory, a direct reference to a class defined later in the file would fail.
- The identity map guarantees that `task.parent.children[0]` resolves to the same object reference as `task` (referential equality).
- Recursive trees (like a subtask hierarchy) work out of the box, but be careful with recursive rendering in React -- use depth limits or lazy loading to avoid infinite loops.

## Complete example: Task, Project, and Team

Here is a realistic domain model with all relationship types:

```ts
import {
  Model,
  ClientModel,
  Property,
  Reference,
  ManyToOne,
  OneToMany,
  BackReference,
  ReferenceArray,
  ReferenceCollection,
} from "@stratasync/core";

// ---- Team ----

@ClientModel("Team", { loadStrategy: "instant" })
export class Team extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  // One team has many projects
  @OneToMany({ foreignKey: "teamId" })
  declare projects: unknown[];

  // One team has many members (unordered)
  @ReferenceCollection({ foreignKey: "teamId" })
  declare members: unknown[];
}

// ---- Project ----

@ClientModel("Project", { loadStrategy: "instant" })
export class Project extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  @Property()
  declare description: string;

  // Many projects belong to one team
  @ManyToOne("Team", "projects", { foreignKey: "teamId" })
  declare team: unknown;

  @Property()
  declare teamId: string;

  // One project has many tasks
  @OneToMany({ foreignKey: "projectId" })
  declare tasks: unknown[];

  // Project lead (single reference)
  @Reference(() => User, undefined, { foreignKey: "leadId" })
  declare lead: unknown;

  @Property()
  declare leadId: string | null;
}

// ---- User ----

@ClientModel("User", { loadStrategy: "instant" })
export class User extends Model {
  @Property()
  declare id: string;

  @Property()
  declare name: string;

  @Property()
  declare email: string;

  // User belongs to a team
  @ManyToOne("Team", "members", { foreignKey: "teamId" })
  declare team: unknown;

  @Property()
  declare teamId: string;

  // Back-references for tasks assigned to this user
  @BackReference({ foreignKey: "assigneeId" })
  declare assignedTasks: unknown[];

  // Back-references for tasks created by this user
  @BackReference({ foreignKey: "creatorId" })
  declare createdTasks: unknown[];
}

// ---- Task ----

@ClientModel("Task", { loadStrategy: "instant" })
export class Task extends Model {
  @Property()
  declare id: string;

  @Property()
  declare title: string;

  @Property()
  declare description: string;

  @Property()
  declare status: string;

  @Property()
  declare priority: string;

  // Many tasks belong to one project
  @ManyToOne("Project", "tasks", { foreignKey: "projectId" })
  declare project: unknown;

  @Property()
  declare projectId: string;

  // Task assigned to a user
  @ManyToOne("User", "assignedTasks", { foreignKey: "assigneeId" })
  declare assignee: unknown;

  @Property()
  declare assigneeId: string | null;

  // Task created by a user
  @ManyToOne("User", "createdTasks", { foreignKey: "creatorId" })
  declare creator: unknown;

  @Property()
  declare creatorId: string;

  // Self-referencing: parent task
  @Reference(() => Task, undefined, { foreignKey: "parentId" })
  declare parent: unknown;

  @Property()
  declare parentId: string | null;

  // Sub-tasks
  @OneToMany({ foreignKey: "parentId" })
  declare children: unknown[];

  // Ordered list of label IDs
  @ReferenceArray()
  declare labels: unknown[];

  @Property()
  declare labelIds: string[];

  @Property()
  declare createdAt: string;

  @Property()
  declare updatedAt: string;
}
```

### Navigating the relationship graph

```ts
// Get a project and traverse relationships
const project = client.getCached("Project", projectId);

// Navigate to team
const team = project.team; // Resolves via identity map

// Get all tasks in the project
const tasks = project.tasks; // LazyCollection

// Get the assignee of an task
const task = tasks[0];
const assignee = task.assignee; // Resolves via identity map

// Navigate back to the user's team
const userTeam = assignee.team; // Same team object (referential equality)
```
